# Механизм работы офлайн-режима и синхронизации данных

В целевой архитектуре розничной сети VitaStore переход на тонкий клиент Web POS требует надежного механизма обеспечения автономной работы касс при сбоях связи, а также корректного слияния данных после восстановления соединения.

## Пояснение

### 1. Автономная работа (Offline-mode)
При разрыве интернет-соединения кассовое приложение Web POS автоматически переключается в автономный режим. Продажа товаров продолжается благодаря локальному кэшированию справочника номенклатуры и цен в браузере с использованием СУБД IndexedDB (или WebAssembly-версии SQLite). Web POS общается с торговым оборудованием через локально запущенную службу HAL (Hardware Abstraction Layer). HAL осуществляет отправку команд на фискальный регистратор по USB/COM-порту для печати физического чека и записи транзакции в фискальный накопитель, что позволяет полностью соблюдать законодательство в оффлайне. В целях безопасности операции лояльности ограничиваются: накопление баллов разрешено (номер карты сохраняется в структуре оффлайн-чека), но списание баллов блокируется для предотвращения мошенничества и повторных трат (double-spending). Оформленные чеки сохраняются в локальной очереди транзакций IndexedDB со статусом `pending_sync`.

### 2. Синхронизация данных при восстановлении связи
Служба фоновой синхронизации в Web POS непрерывно опрашивает API Gateway с помощью механизма Heartbeat (периодические легкие PING-запросы). При восстановлении связи статус приложения меняется на онлайн, и запускается фоновый экспорт накопленных транзакций. Чеки передаются пакетами (batches) по протоколу HTTPS REST на облачный бэкенд Cloud POS. Для гарантии надежности доставки используется уникальный идентификатор транзакции (`transaction_id` в формате UUID v4), генерируемый кассой в момент пробития чека. Облачный бэкенд использует этот UUID в качестве ключа дедупликации (идемпотентность API): при повторной отправке одного и того же пакета из-за нестабильного соединения дубликаты транзакций мгновенно отсекаются на уровне СУБД бэкенда, исключая задвоение продаж.

### 3. Стратегия разрешения конфликтов
Для слияния данных после оффлайна применяется гибридная стратегия разрешения коллизий. Для финансовых транзакций и фактов продаж действует правило **«Последняя запись побеждает» (Last-Write-Wins, LWW)** на основе временных меток (timestamp), установленных локальными часами фискального накопителя кассы, так как касса является единственным источником правды для факта совершения сделки. Для товарного учета (остатков на складе) применяется **арифметическое delta-based слияние**: вместо перезаписи абсолютного значения остатка на сервере, облачный сервис инвентаризации вычитает из текущего серверного баланса дельту (количество проданного товара в оффлайне). Это предотвращает перезапись изменений остатков, которые могли произойти на сервере во время оффлайна кассы (например, при проведении приходов товаров). В случае возникновения критических неразрешимых коллизий (например, отрицательный остаток по маркированному товару), система генерирует алерт для ручного разбора ИТ-дирекцией и товароведом магазина.
